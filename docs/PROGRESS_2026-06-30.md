# Progress Report — 2026-06-30

Single-day summary of the 7-round login-bug fix and the 5-bug
production smoke that followed. Captures what was done, why, what
remains, and the cross-cutting follow-up that should be the
**next epic** for any agent continuing the work.

## TL;DR

- **Login flow** (POST /api/v1/auth/refresh) was broken in 4
  different ways. Took 7 rounds to fully fix.
- After the login fix, ran a public-API smoke and found **5
  production bugs** that the migration chain / CI did not catch.
- All 5 are now fixed. The smoke is documented in
  `docs/SMOKE_PROD_2026-06-30.md`.
- 4 new Lessons added to `docs/LESSONS.md` (15-18 from the
  login rounds, 19-20 from the smoke).
- **Main remaining follow-up**: `alembic upgrade head` is
  missing from Render's startCommand, so every migration since
  0001 has been a no-op in production. This is the **root cause**
  of the schema-drift class of bug (Bugs 1, 5) and will recur if
  not fixed.

## Login flow — 7 rounds of fixes

| Round | Symptom | Root cause | Fix |
|---|---|---|---|
| 1 | `/positions` page empty | Pydantic `created_at: datetime` (no `= None`) + legacy NULL rows in DB | `created_at: datetime | None = None` + migration 0037 backfill |
| 2 | `refresh_access_token` crashed with `TypeError: Object of type UUID is not JSON serializable` | PyJWT 2.x uses stdlib `json.dumps`; UUID is not JSON-serializable | `app/core/auth.py::_json_safe_jwt_payload` normalises UUID → str at the encode boundary |
| 3 | `/refresh` returned 200 but frontend `_refresh()` did not call `setAuth()` | Backend `TokenResponse` did not include `user` field; frontend guard required `data.user` | Added `user: UserResponse` to `TokenResponse` |
| 4 | After R3, /refresh still didn't work; user still bounced to /login | R3 fix used wrong shape (`UserResponse` lacks `role`, `tenant`, `full_name` that frontend AuthUser needs) | Round 7 fix below |
| 5 | After R2+alembic, /refresh still 401 with `InvalidTokenError` | R2's `_json_safe_jwt_payload` converted `exp`/`iat` from datetime to isoformat strings; PyJWT `verify_exp` calls `datetime.fromtimestamp(exp)` on these and crashes | `exp/iat/nbf` set as `int(...timestamp())`; `_json_safe_jwt_payload` no longer touches datetime |
| 5b | Deploy exited status 1 (IndentationError in /refresh) | R5 edit left a stray block from R4 that wasn't matched by oldString | Removed the stray lines |
| 6 | Still 401, no traceback to learn from | R3's `except Exception: raise HTTPException(401)` swallowed the type | Added catch-all `print(traceback)` in `decode_token` and `/refresh` |
| 7 | After R6 logs, identified the real cause (R5's wrong shape) | R3 returned `UserResponse` shape; frontend needs `AuthUser` shape | New `build_user_payload(db, user, telegram_id)` helper, single source of truth for AuthUser shape; used by both `/check-code` and `/refresh` |
| 7b | Logout 401 after access-token expiry (1h TTL) | `get_current_user` dep on `/logout` rejected the expired access-token before logout could run | Logout now decodes the refresh-token directly (30-day TTL); `get_current_user` removed |

**Final state:** Login flow end-to-end works. /refresh returns
`user` in the right shape. /logout is functional. Render logs
show check-code polling 200 every 5s for the demo tenant.

## Production smoke — 5 bugs found and fixed

After the login fix, ran `Invoke-RestMethod` calls against
`/api/v1/*` (142 endpoints discovered via OpenAPI). Used the
public `POST /auth/demo-login` to obtain a JWT and exercise
authed endpoints. Five prod bugs surfaced.

| # | Endpoint | Symptom | Fix | Commit |
|---|---|---|---|---|
| 1 | `GET /certificates/verify/{number}` | 500 (`column certificates.certificate_number does not exist`) | Manual DB reconcile (rename `cert_number`→`certificate_number`, add `expires_at`/`pdf_path`/`metadata`); migration 0038 makes it idempotent | 936137b, fc318b4 |
| 2 | `POST /auth/register` | 500 (`UserCreate object has no attribute tenant_id`) | Use `uuid4()` for new tenant id; never trust client `tenant_id` | 936137b |
| 3 | `POST /telegram/webhook` | 200 on empty body (open relay) | Required `X-Telegram-Bot-Api-Secret-Token` header; fail-closed (404) if secret not configured. Secret set in Render env, webhook re-registered with Telegram | 936137b, fc318b4 + env update + setWebhook |
| 4 | `POST /api/v1/positions` | 500 (`AttributeError: 'str' object has no attribute '_sa_instance_state'`) | `Position.department` was declared twice (Column + relationship). Renamed relationship to `department_obj` | d39e106 |
| 5 | `POST /api/v1/positions/{id}/courses` | 500 (`null value in column 'tenant_id' of relation 'position_courses' violates not-null constraint`) | Added `tenant_id` field to `PositionCourse` model; passed `user.tenant_id` to all three insert sites | 8638ea3, 942f3ce |

All 5 fixes verified end-to-end via the public API after deploy.
Bug 5 was specifically verified by direct `asyncpg` query that
showed a new row in `position_courses` with the correct
`tenant_id`.

## Cross-cutting follow-up (the next epic)

**`alembic upgrade head` is missing from Render's startCommand.**

The `render.yaml` and the Dashboard both show
`uvicorn app.main:app --host 0.0.0.0 --port $PORT` as the
startCommand. There is no `alembic upgrade head && ...` prefix.
Result: every Alembic migration since 0001 has been a **no-op**
in production. The schema drift that caused Bugs 1 and 5 (and
possibly 4) only exists because the DB was originally created
by `Base.metadata.create_all()` at first uvicorn boot, then
someone hand-edited columns.

**Action:** add `alembic upgrade head &&` to the startCommand.
This is a 1-line config change but a **large** systemic impact:
every future migration will apply automatically and the schema
drift class of bug will not recur.

**Verification:** after the change, deploy once and check
`alembic_version` table in the DB — it should advance from
`0036` (current) to whatever the latest revision is. The
manual `Base.metadata.create_all` set-up is now redundant and
can be removed from the boot path.

## Why 5 bugs existed in production

Three of the five (1, 3, 5) are the same root cause: **prod DB
and code drifted because migrations don't run on deploy**. One
(2) is a register endpoint that was never smoke-tested. One
(4) is a model definition that was never checked against the
DB constraints.

The smoke test (this document) caught all five because the
smoke calls real endpoints with real auth, not mocked unit
tests. **Recommendation:** make the smoke script a CI step
that runs after every prod deploy. The script is short
(20-30 lines of `Invoke-RestMethod` calls) and would have
caught every one of these bugs.

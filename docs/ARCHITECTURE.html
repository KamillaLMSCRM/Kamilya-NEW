🎓 SCORM Agents — Архитектура

Мульти-агентная RAG-система, превращающая корпоративные документы (PDF / DOCX / PPTX) в готовые интерактивные SCORM 1.2 курсы для LMS (Chamilo и др.). Два интерфейса — CLI и Telegram-бот — используют одно и то же ядро конвейера.

Пять стадий: Ingest (Docling → ChromaDB) → Design (Architect Agent) → Generate (Writer + RAG) → Assess (вопросы + Question Bank) → Package (HTML → SCORM ZIP).

Python 3.12+ LangGraph · LangChain vLLM (Qwen3.5-122B) ChromaDB Docling aiogram 3 SQLite (aiosqlite) Typer CLI Jinja2 Langfuse i18n: en / ru / kk

Технологический стек и внешние сервисы

Система зависит от трёх vLLM-инстансов (OpenAI-совместимый API), запущенных отдельно. Это внешние сетевые зависимости — если они недоступны, падает почти всё.

Две точки входа над одним ядром

🖥 CLI — cli.py (Typer)

Интерактивный запуск из терминала. Команды:

  • create — полный конвейер: ingest → интерактивное утверждение структуры (y/r/e) → generate → package
  • package — пересборка ZIP из сохранённых JSON (без AI)
  • health — проверка доступности vLLM-сервисов

Прогресс через ConsoleReporter. Структура утверждается человеком в цикле _interactive_architect_loop().

🤖 Telegram Bot — bot/main.py (aiogram 3)

Тот же конвейер через диалог. FSM-флоу: регистрация → загрузка → язык → конфиг → guidance → очередь → ZIP.

  • Последовательная asyncio.Queue (1 задача за раз → не перегружать vLLM)
  • pipeline_runner.py вызывает то же ядро: ingest_documents → run_architect → process_course_with_modules → package_course
  • Прогресс через TelegramReporter (редактирует одно сообщение)
⚠ Важно

CLI и бот — это два разных оркестратора над общим ядром. CLI использует run_pipeline() (legacy, плоский), бот — process_course_with_modules() (module sub-graph + Question Bank + тесты). Изменение ядра влияет на оба, но проявляется по-разному.

① Ingestion — обработка документов

Пакет src/scorm_agents/ingestion/. Точка входа — ingest_documents(docs_dir, settings) → IngestionResult. Превращает документы в чанки в ChromaDB плюс образовательные резюме на диске.

② Architect Agent — проектирование структуры

Пакет src/scorm_agents/architect/. ReAct-агент (LangGraph create_react_agent) исследует документы инструментами и выдаёт CourseStructure в JSON.

Точка входа

run_architect(settings, on_message, doc_ids, goals, course_hours, num_sections, language, guidance)

Строит локализованный system-prompt, гоняет ReAct-цикл, парсит финальный AI-ответ (```json блок, sanitize) → CourseStructure. Guardrails: architect_recursion_limit=30, max_chapters_per_doc=5.

5 инструментов (tools.py)

  • list_documents() — все doc_id/имена
  • get_document_summary(doc_id) — профиль из summaries
  • get_document_toc(doc_id) — иерархия TOC
  • get_chapter_text(doc_id, title) — текст главы (≤8000, лимит на док)
  • search_documents(query, doc_id?) — семантический поиск (embed → VectorStore)

Схема результата — CourseStructure (architect/schema.py)

CourseStructure( title: str, description: str, modules: list[Module] # Module(title, lessons[]) └ lessons: list[Lesson] ├ title: str ├ objectives: list[LearningObjective] # .text ├ source_doc_ids: list[str] # какие документы (ADR-042 роутинг) └ relevant_headings: list[str] # подсказки для RAG Writer'а ) # to_json() · from_json() · from_yaml() — сохраняется в course_structure.json
Связь с Writer

source_doc_ids и relevant_headings каждого урока пробрасываются в Writer и управляют тем, какие документы и заголовки участвуют в RAG-выборке. Architect фактически программирует Writer.

③ Writer Agent — генерация контента (RAG)

Пакет src/scorm_agents/writer/. Детерминированный 3-шаговый конвейер (ADR-057), не ReAct — ровно 1 LLM-вызов на урок. Запросы строятся детерминированно.

Точка входа — write_lesson(...) → LessonContent

1

Queries

_generate_queries() из title + objectives + headings (без LLM)

→
2

Retrieve

_retrieve_and_rerank() multi-query: embed → ChromaDB → reranker ИЛИ cosine-gate → top-N

→
3

Generate

_generate_from_chunks() — 1 LLM-вызов → Markdown

RAG-детали

  • Reranker путь (если включён): embed с Instruct-префиксом → дедуп чанков → rerank по title → top_n
  • Cosine путь (по умолч.): фильтр по порогу retrieval_similarity_threshold=0.45, fallback на top-N если всё отфильтровалось
  • Heading-контекст (ADR-041): чанки форматируются как [Context: h1 > h2]\ntext
  • Small-doc bypass (ADR-P1): для малых документов retrieval пропускается, передаются все чанки
  • Anti-repetition (ADR-P3): sibling_lessons в промпте, чтобы уроки не дублировались
  • Hard timeout: lesson_content_timeout=1200s → зависший урок становится failed, не блокирует курс

Схемы — writer/schema.py

LessonContent( title, objectives: list[str], content: str # Markdown source_chunks: list[str] # цитаты RAG ) ModuleContent(title, lessons[]) CourseContent(title, description, modules: list[ModuleContent]) # → course_content.json

④ Assessment — генерация вопросов

Пакет src/scorm_agents/assessment/. Грунтованная генерация MCQ / True-False / Matching прямо из контента урока. vLLM guided_json гарантирует валидный JSON.

Два режима

  • Простой — generate_lesson_assessment() (agent.py): 1 LLM-вызов → LessonAssessment. Используется CLI/legacy.
  • 3-pass — generate_lesson_questions_3pass() (generator.py): N проходов (assessment_passes, по умолч. 1) с накоплением контекста, дедуп, финальный проход с оценкой качества (1–5). Результат → Question Bank.

Жёсткий JSON-парсинг: удаление комментариев, починка кавычек/скобок, trailing commas, control-chars.

Схемы — assessment/schema.py

LessonAssessment( lesson_title, mcq: list[MCQQuestion] true_false: list[TrueFalseQuestion] matching: list[MatchingQuestion] ) CourseAssessment(assessments[]) # каждый Q имеет quality_score: float # → course_assessment.json
Связь с Question Bank

3-pass возвращает (list[Question], mcq_options_map, matching_pairs_map) → enrich_and_store_questions() сохраняет в SQLite. Из банка потом собираются модульные и финальный тесты.

⑤ Question Bank — банк вопросов (SQLite)

Пакет src/scorm_agents/question_bank/. Хранит вопросы, отбирает их в тесты с воспроизводимым seed'ом и дедупликацией. Async через aiosqlite, WAL-режим.

Схема БД

questions (PK question_id) ├ lesson_id, module_id, course_id # ⚠ scoping-ключи ├ question_type CHECK(mcq|true_false|matching) ├ difficulty CHECK(easy|medium|hard) ├ quality_score REAL[0-5], question_text, correct_answer ├ used_in_tests INT, generation_pass INT[1-3] └ language TEXT DEFAULT 'en' # migration #2 mcq_options (PK option_id, FK question_id CASCADE) → text, is_correct, order matching_pairs (PK pair_id, FK question_id CASCADE) → left, right, order schema_version (version INT) # миграции

db.py

storage.py

enrich_and_store_questions() — обогащение метаданными + транзакционный батч-save (rollback при ошибке)

selection.py

stratified_select · get_seed · select_module_questions (10: 5+3+2) · select_final_questions (20, квоты по урокам)

🔥 Критичный нюанс — course_id scoping

module_id переиспользуется между курсами (например, "module_0" существует в каждом курсе). Поэтому в select_*_questions обязательно передавать course_id, иначе тесты соберут чужие вопросы. Это частый источник «странных» багов. Seed: "auto" = SHA256(course_id) → детерминированный отбор.

⑥ SCORM Packaging — сборка ZIP

⑦ LangGraph оркестрация — ядро параллелизма

Пакет src/scorm_agents/pipeline/. Связывает Writer + Assessment + Question Bank через Send API (fan-out / fan-in). Два графа.

Граф A — core.py (legacy, CLI)

run_pipeline() → (CourseContent, CourseAssessment)

START → distribute_lessons # fan-out: Send на урок → process_lesson ×N # параллельно write_lesson + generate_lesson_assessment → assemble # fan-in (operator.add) → END

Граф B — module_graph.py (бот, D-09)

process_course_with_modules() → dict

для каждого модуля (asyncio.gather): run_module_subgraph: START → distribute_module_lessons # fan-out → process_lesson_for_module ×N write_lesson + 3pass → Question Bank → generate_module_questions # барьер fan-in → generate_module_test # SELECT из QB → END

Состояния (TypedDict)

StateКлючевые поляНазначение
PipelineStatecourse_structure, doc_ids, lesson_results: Annotated[list, operator.add], languagefan-in reducer склеивает результаты Send
LessonStatelesson_title, objectives, module_index, lesson_index, doc_ids, languageминимальные данные для параллельного урока
ModuleStatemodule_index, lessons[LessonInfo], course_id, question_bank_ready, module_testкоординация модуля + Question Bank scoping
⚠ Тонкие места оркестрации
  • Per-lesson Langfuse handler + отключение callbacks во время Send — обход дедлока OpenTelemetry в параллельных корутинах.
  • Hard timeout asyncio.wait_for на урок → stub-контент вместо блокировки всего курса.
  • Annotated[list, operator.add] — единственный механизм сборки результатов; менять тип поля рискованно.

Команда package как контракт-тест

CLI-команда package собирает ZIP только из трёх JSON. Значит, эти JSON — стабильный промежуточный контракт: если они валидны, упаковка работает без AI и моделей. Удобно для отладки «что сломалось».

3 независимых хранилища состояния

ChromaDB (RAG-память) · bot.db (сессии Telegram) · question_bank.db (вопросы). Они не связаны транзакционно — рассинхрон между ними (например, ChromaDB сброшен, а bot.db ссылается на старые job) может давать неожиданные ошибки.

⚠ Точки связности — «где что ломается»

Главная причина каскадных поломок — несколько неявных контрактов между подсистемами. Вот они, по убыванию риска. Меняя что-то в одном модуле, проверь связанные.

1. Три JSON-схемы
CourseStructure / CourseContent / CourseAssessment сериализуются между стадиями и сохраняются на диск. Переименование/удаление поля ломает from_dict/from_json в Packager и боте. Перед изменением прогони CLI-команду package на сохранённых JSON.
2. course_id в Question Bank
module_id ("module_0") переиспользуется между курсами. Если в select_module_questions/select_final_questions не передать course_id — тест соберёт чужие вопросы. Самый коварный «тихий» баг.
3. doc_ids роутинг (ADR-042)
Architect кладёт source_doc_ids в каждый урок; они доходят до Writer'а и ограничивают RAG-выборку. Несовпадение doc_id (stem файла) между ingestion и структурой → урок пишется по пустому/чужому контексту. doc_id строится из имени файла — переименование документа ломает связь.
4. Язык (language)
Параметр language пробрасывается сквозь Architect → Writer → Assessment → Packager → renderer/manifest и UI-строки. Пропуск проброса в одном узле → курс на смешанных языках.
5. Два оркестратора
CLI = run_pipeline (плоский, без QB-тестов), Бот = process_course_with_modules (module sub-graph + Question Bank). Правка в writer/assessment отражается в обоих, но тесты QB есть только во втором. Легко «починить» в CLI и не заметить регресс в боте.
6. LangGraph state reducer
lesson_results: Annotated[list, operator.add] — механизм fan-in. Изменение типа поля или структуры результата process_lesson ломает assemble(). Плюс хрупкий обход дедлока OpenTelemetry (per-lesson Langfuse handler + отключение callbacks).
7. Внешние vLLM-сервисы
LLM/Embedding/Reranker — внешние по сети. Недоступность → LLMError почти на любой стадии. health проверяет связность. Reranker по умолчанию выключен (reranker_enabled=False).
8. Миграции SQLite
У bot.db и question_bank.db свои migrations.py / schema_version. Добавление колонки требует миграции, иначе старая БД падает на новых запросах (напр. колонка language).
9. JSON от LLM
Architect и Assessment парсят сырой LLM-вывод. Есть тяжёлый sanitizer (комментарии, кавычки, скобки, trailing commas). При смене модели/промпта формат вывода может «поплыть» и обойти ремонт.
10. Jinja2 шаблоны ↔ данные
lesson.html.j2 / test.html.j2 ожидают конкретные поля. Изменение схемы assessment без правки шаблона → пустые/битые карточки вопросов в SCORM.
💡 Рекомендация по стабильности

Перед изменением общих схем или Question Bank: (1) сохрани эталонные JSON курса, (2) после правки прогони scorm-agents package на них — это быстрый изолированный тест Packager'а без моделей. (3) Прогони uv run pytest (есть тесты pipeline/, scorm/, question_bank/, bot/, e2e/). (4) Проверь и CLI, и бот-путь — у них разные оркестраторы.

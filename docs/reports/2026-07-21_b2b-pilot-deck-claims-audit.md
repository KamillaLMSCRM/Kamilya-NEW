## Метод проверки

Для каждого заявления различались четыре уровня доказательств:

1. наличие реализации в коде;
2. автоматические тесты;
3. доступность текущего production API и согласованность production-схемы;
4. ранее зафиксированный controlled production E2E.

Наличие страницы или endpoint само по себе не считалось доказательством рабочего пользовательского сценария.

## Вывод

Презентацию можно использовать для квалификации B2B-пилота. В ней нет прежних опасных обещаний про compliance, ЭЦП, обязательные журналы или автоматическое качество AI.

Главная недоказанная формулировка не текстовая, а подразумеваемая: показанные этапы выглядят как один уже проверенный сквозной сценарий. На деле production-доказательства состоят из двух тестов: AI generation smoke и отдельного native learner E2E. Перед демонстрацией первому клиенту нужен один новый изолированный E2E, который свяжет весь путь без разрыва:

`документ → успешная индексация → AI-курс и тест → approve методологом → publish → назначение 3 learners → прохождение → passed quiz → certificate → training log → cleanup`.

До этого корректно говорить: **«все этапы реализованы и отдельно подтверждены в production»**, но не **«весь показанный путь регулярно проходит без сбоев как единый сценарий»**.

## Обязательные действия перед клиентским показом

1. Провести единый controlled E2E по цепочке выше на текущем commit `a91be55`.
2. Очистить или исправить 4 исторических `published/pending` AI-курса.
3. Повышать обязательный coverage baseline выше текущих 50% отдельными тестами AI router/pipeline, users/invitations и tenant registration.
4. Не добавлять в презентацию сроки генерации, SLA, процент успешности AI или юридическую значимость до появления измерений и соответствующих оснований.

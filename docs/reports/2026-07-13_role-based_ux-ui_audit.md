# Ролевой UX/UI-аудит Kamilya LMS

Дата: 2026-07-13

## 1. Вывод для руководства

Основная проблема интерфейса не в отсутствии функций, а в смешении трёх разных зон ответственности:

1. управление компанией и людьми;
2. методология обучения и назначение траекторий;
3. прохождение обучения.

Backend уже в значительной степени разделяет эти зоны, но sidebar и названия экранов местами показывают функции шире, чем фактические права API. Это создаёт ложные ожидания: пользователь видит пункт, открывает его и получает пустой экран, 403 или непонятную бизнес-логику.

Целевая модель:

| Роль | Главный результат работы |
|---|---|
| Superadmin | Поддержать платформу и тенантов, не управлять обучением внутри компании |
| Администратор тенанта / HR | Настроить компанию, лимиты, пользователей команды, киоски, сертификаты, контроль результатов |
| Методолог | Создать и проверить обучение, сформировать траектории, когорты, назначения и тесты |
| Teacher | Тот же learning-контур в текущей модели; желательно оставить как технический alias методолога |
| Обучающийся | Увидеть свои курсы, пройти уроки/тесты, получить сертификат |
| Оператор киоска | Быстро идентифицировать сотрудника и запустить назначенное обучение без доступа к кабинету компании |

## 2. Фактическая модель ролей

Роли в API: `superadmin`, `admin`, `org_admin`, `methodologist`, `teacher`, `student`. В `apps/api/app/core/auth.py` `teacher` фактически используется как learning-role alias, но frontend не объясняет эту связь. Это источник терминологической путаницы: в одной части продукта пользователь называется методологом, в другой — teacher.

Ключевые фактические ограничения:

- прямое назначение курса выполняют `methodologist` и `teacher`;
- назначение теста в API относится к learning-контру, но sidebar сейчас показывает этот пункт администратору;
- создание обычного пользователя доступно администратору, однако создание `student` намеренно исключено: обучающиеся создаются через student-provisioning, приглашение, Telegram или kiosk-flow;
- журнал обучения доступен `admin`, `org_admin`, `methodologist`, `superadmin`, но не `teacher`;
- шаблон сертификата в API изменяют только `admin` и `org_admin`, хотя sidebar показывает его методологу и teacher;
- импорт штатки и структура персонала доступны нескольким ролям, но UI не объясняет, кто владеет справочником людей, а кто только использует его для назначения;
- superadmin имеет отдельный route guard и не должен попадать в tenant-scoped страницы.

## 3. Главные несоответствия

### P0 — исправить до расширения первого тенанта

| Проблема | Доказательство | Риск | Решение |
|---|---|---|---|
| Курсы и тесты назначаются разными ролями | `enrollments/router.py` ограничивает прямое назначение курса `methodologist`/`teacher`, а `quizzes/assignment_router.py` разрешает `admin`, `org_admin`, `teacher` | HR может назначить тест, но не курс; методолог может назначить курс, но не видит этот пункт в текущем sidebar | Зафиксировать одну policy-модель: либо оба действия принадлежат методологу, либо оба принадлежат admin+methodologist. Для продукта рекомендую admin отвечает за контроль, методолог — за назначения обучения |
| Шаблон сертификата показывается методологу/teacher | `certificates/router.py` разрешает только `admin`, `org_admin` | UI обещает действие, которое API не разрешает | Оставить редактирование HR/admin; методологу дать только просмотр/preview в карточке курса |
| Методолог и teacher визуально не разведены | Backend принимает оба имени, frontend использует оба без объяснения | Непонятно, кого приглашать и кто отвечает за результат | На UI использовать «Методолог», `teacher` оставить совместимостью в API и миграции ролей |
| `/admin/enrollments` остаётся доступным по URL | Страница существует, но direct enrollment API принадлежит learning-role | Сохраняется старый сценарий, который пользователь ранее считал ошибочным | Сделать redirect на `/assignments` или удалить страницу после проверки bookmark compatibility |
| Tenant admin видит «Контент» слишком широко | Курсы, AI, quizzes и документы показываются admin/org_admin, хотя часть операций является методологической | HR получает перегруженное меню и не понимает, где заканчивается его роль | Разделить режимы: admin видит обзор/лимиты/результаты, методолог — authoring и delivery; временно оставить admin только при явной бизнес-потребности |

### P1 — исправить в ближайшем UX-потоке

| Проблема | Доказательство | Предложение |
|---|---|---|
| Администратор видит `learning paths`, `cohorts`, `competencies`, `surveys` вместе с authoring | Sidebar и backend разрешают эти функции сразу нескольким manager-ролям | Ввести подзоны «Программы обучения», «Аудитория», «Контроль» с role-aware составом |
| `Штатное расписание` содержит импорт, структуру, привязки и курсы компании | Один экран перегружен разными задачами | Переименовать в «Персонал и назначения», оставить tabs, но показывать первый tab в зависимости от роли |
| `Управление пользователями` фактически управляет командой, не обучающимися | `/admin/team` исключает students; students создаются другим flow | Переименовать в «Команда и администраторы», рядом дать явную кнопку «Добавить обучающихся» через приглашение/штатку |
| `Админ` не говорит, что это панель компании | `/admin` — dashboard с лимитами, статистикой и экспортами | Название «Панель компании» или «Контроль компании», а «Настройки» оставить отдельно |
| `Интеграции` доступны superadmin и tenant admin в одном пункте | Это разные уровни управления | Superadmin: provider keys и глобальные health; tenant admin: Resend/Telegram/уведомления компании |
| Student navigation почти отсутствует у org_admin, но добавлена искусственно | `my-courses`, `my-quizzes`, `certificates` показываются `org_admin` | Убрать learner-mode из HR-профиля или сделать явный переключатель «Просмотр как обучающийся» |

### P2 — качество и последовательность

- заменить `teacher` в пользовательских текстах на «Методолог»;
- унифицировать `Курсы`, `Тесты`, `Назначения`, `Когорты` в одной терминологии;
- для каждой 403 показывать объяснение и ссылку на владельца функции, а не общий error state;
- скрывать пустые разделы до первого созданного объекта, но показывать понятный CTA;
- добавить role-based breadcrumbs: `Компания -> Обучение -> Назначения`;
- разделить platform navigation и tenant navigation визуально, как уже сделано для superadmin.

## 4. Аудит по ролям

### 4.1 Superadmin

**Должен видеть:** список тенантов, карточку тенанта, platform providers, impersonation/support tools, platform audit.

**Не должен видеть:** AI generation, courses, staff, assignments, training log конкретного тенанта в обычной навигации.

**Текущее состояние:** `Layout.tsx` ограничивает superadmin префиксами `/admin/super`, `/admin/providers`, `/superadmin`; это правильно. Sidebar также показывает tenant integrations superadmin, что может привести к ложному ожиданию tenant-context.

**Предложение:** оставить только «Тенанты», «Провайдеры AI», «Аудит платформы». Impersonation запускать только с карточки конкретного тенанта, с явным banner «Вы действуете от имени компании X» и кнопкой выхода из impersonation.

### 4.2 Администратор тенанта / HR

**Должен видеть:** панель компании, лимиты trial/тарифа, команда и администраторы, обучающиеся/штатка, киоски, сертификатный шаблон, журнал обучения, tenant integrations, экспорт.

**Не должен быть владельцем:** ручные назначения курса и теста, редактирование учебного контента, AI-генерация, методологические траектории.

**Текущее состояние:** admin получает AI, courses, quizzes, documents, staff structure, positions и test assignment в sidebar. Часть этого доступа разрешена API, но обучение и управление компанией смешаны в одном меню. При этом admin не может выполнить симметричное прямое назначение курса через enrollment API.

**Предложение навигации:**

1. Панель компании
2. Люди и доступы
3. Обучающиеся и штатка
4. Киоски
5. Результаты и журнал
6. Сертификаты
7. Настройки и уведомления

В «Люди и доступы» отдельно показать: администраторы/методологи и обучающиеся. Не использовать один экран «Пользователи» для всех типов.

### 4.3 Методолог

**Должен видеть:** документы, AI-генерацию, курсы, конструктор тестов, learning paths, cohorts, competencies, штатка в режиме выбора аудитории, назначения курсов/тестов, приглашения обучающихся, журнал прогресса.

**Не должен редактировать:** тариф, лимиты, tenant integrations, список администраторов, provider keys.

**Текущее состояние:** методолог имеет доступ к API назначения и импорту, но sidebar не показывает AI/courses/documents для роли `methodologist`; одновременно сертификатный шаблон и журнал видны шире, чем нужно.

**Предложение навигации:**

1. Рабочая область методолога
2. Источники и документы
3. Курсы и тесты
4. Программы обучения
5. Группы и назначения
6. Приглашения обучающихся
7. Прогресс обучения

Для первого тенанта это наиболее важный UX-приоритет: методолог должен пройти путь «документ -> курс -> тест -> группа/сотрудники -> назначение -> контроль» без переходов в `/admin/*`.

### 4.4 Teacher

Сейчас это технически отдельная роль с почти теми же правами, что методолог. Для продукта две разные UX-модели не оправданы, пока нет различий в ответственности.

**Предложение:** показывать пользователю роль «Методолог» и хранить `teacher` как совместимый backend alias. Если позднее появится преподаватель, которому разрешено только вести занятия и смотреть свою группу, тогда выделить роль отдельной политикой, а не текущим широким доступом.

### 4.5 Обучающийся

**Должен видеть:** мои курсы, текущий прогресс, тесты, сертификаты, AI-assistant внутри курса, профиль и выход.

**Не должен видеть:** sidebar authoring, штатка, назначения, журнал компании, admin, cohorts и platform routes.

**Текущее состояние:** student landing корректно ведёт на `/student`, но часть общих страниц (`learning paths`, `cohorts`, `surveys`) включена в sidebar для student, хотя это не соответствует минимальному learner-flow.

**Предложение:** сделать learner shell без административного sidebar. На первом экране показывать «Продолжить обучение», дедлайны, тесты и сертификаты. Общие опросы должны появляться только как задания, а не как отдельный управленческий раздел.

### 4.6 Оператор киоска

Это не обычная роль кабинета. Киоск должен работать по короткому токену и иметь отдельный shell: идентификация сотрудника, список назначенных курсов, запуск, завершение, печать/скачивание результата.

**Предложение:** не показывать kiosk в общей навигации обучающегося; администратору оставить управление устройствами и токенами. В kiosk-flow добавить авто-logout, очистку локального состояния и видимое подтверждение, что предыдущий сотрудник вышел.

## 6. План изменений

### P0.1 — убрать ложные действия

1. Согласовать владельца назначения курса и теста, затем привести sidebar и API к одной policy-модели.
2. Скрыть редактирование certificate template у methodologist/teacher; дать preview.
3. Удалить/redirect `/admin/enrollments` на `/assignments`.
4. Исправить `methodologist`/`teacher` terminology в UI.
5. Добавить понятную 403-страницу с владельцем функции.

### P0.2 — собрать первый рабочий flow методолога

1. Дать методологу видимые пункты AI, documents, courses, quizzes.
2. Создать единый экран «Назначения» с курсами, тестами и cohort apply.
3. На странице staff/штатки разделить «источник аудитории» и «назначение обучения».
4. Добавить CTA после генерации курса: «Проверить тест -> Опубликовать -> Назначить».

### P1 — перестроить оболочки ролей

1. Развести HR/admin shell, methodologist shell и learner shell.
2. Убрать learner shortcuts из org_admin или сделать явный режим «Просмотр как обучающийся».
3. Разделить tenant integrations и platform providers.
4. Пересобрать dashboard под задачу роли, а не под общий список метрик.
5. Добавить role-based Playwright smoke matrix: 6 ролей x ключевые маршруты.

### P2 — полировка

1. Проверить русскую/казахскую терминологию во всех labels и empty states.
2. Согласовать названия маршрутов и breadcrumbs.
3. Добавить доступность: keyboard navigation, focus states, mobile role shells.
4. Скрыть или пометить experimental modules, пока их backend workflow не завершён.

## 7. Критерий готовности

Ролевой UX считается исправленным, когда:

- ни одна роль не видит в sidebar действие, которое её API заведомо запрещает;
- методолог проходит полный learning-flow без `/admin/*` legacy-переходов;
- tenant admin управляет компанией и результатами без необходимости разбираться в authoring-инструментах;
- student видит только обучение и не получает управленческие разделы;
- superadmin не попадает в tenant-scoped shell без явного impersonation;
- для каждой роли есть desktop и mobile smoke-проверки.

## 8. Выполнено после аудита

В рамках согласованной модели внесены первые изменения:

- tenant admin/org_admin убраны из authoring и learner-навигации;
- для admin/org_admin добавлен frontend route guard на прямые переходы к курсам, тестам, документам, штатке, назначениям и приглашениям;
- API-операции создания и изменения курсов, тестов, SCORM, документов, траекторий, когорт, компетенций, опросов и объявлений ограничены методологом/teacher;
- импорт штатки, ручное добавление обучающегося, приглашения и просмотр списка обучающихся переведены в learning-role контур;
- шаблон сертификата, киоски, tenant settings и журнал обучения оставлены tenant admin/org_admin согласно согласованному разделению;
- `teacher` сохранён как технический backend alias методолога.

Следующий обязательный шаг — staging E2E под двумя tenant-аккаунтами: убедиться, что admin получает 403 на learner mutations, а методолог может пройти полный flow `документ -> курс -> тест -> штатка -> назначение`.

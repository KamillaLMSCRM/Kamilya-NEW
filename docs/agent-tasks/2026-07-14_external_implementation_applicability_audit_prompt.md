# Задание внешнему агенту: строгий аудит имплементации и применимости Kamilya LMS

## Роль агента

Ты выполняешь независимый технический аудит проекта Kamilya LMS. Твоя задача — определить не то, насколько хорошо проект описан или выглядит, а насколько его фактическая реализация:

1. соответствует заявленной продуктовой идее;
2. реально применима для первого production-тенанта;
3. актуальна по состоянию текущего checkout и deployed-конфигурации;
4. достаточно целостна по цепочке UI → API → БД → фоновые задачи → storage → уведомления;
5. не содержит ложных обещаний, мёртвых экранов, частично реализованных flows и опасных расхождений между ролями.

Аудит должен быть строгим, доказательным и критическим. Не добавляй комплименты, общие слова и маркетинговую похвалу. Если функция отсутствует, недоступна роли, работает только в happy path, зависит от ручной DB-операции или не подтверждается тестом, помечай её как незавершённую. Не подменяй отсутствие доказательств предположениями.

## Обязательные правила работы

1. Работай в актуальной локальной папке репозитория `KamillaLMSCRM/Kamilya-NEW`, предварительно обнови checkout с удалённого репозитория и зафиксируй commit SHA.
2. Сначала прочитай `AGENTS.md`, затем `PROJECT.md`, `docs/PROJECT-CONTEXT.md`, `docs/PROJECT_INTERNAL_DOCUMENTATION.md`, ADR по RBAC/RLS/auth, текущие отчёты QA и `docs/LESSONS.md`.
3. Документация является гипотезой, а не источником истины. Каждое существенное утверждение сверяй с реализацией.
4. Не раскрывай и не копируй секреты из `.env`, Render, Vercel, Supabase, VPS или GitHub. Не меняй production-данные, миграции, auth, RLS, billing или инфраструктуру без отдельного письменного разрешения.
5. Не исправляй код в рамках этого задания. Если обнаружил проблему, опиши минимальный fix, затронутые файлы, риск и тесты, но оставь изменения в рабочем дереве нетронутыми.
6. Не объявляй flow рабочим по наличию маршрута или кнопки. Нужны трассировка до backend и проверка состояния после перезагрузки страницы.
7. Для каждого вывода указывай доказательство: файл и строку, endpoint, SQL/модель, тест, screenshot/video, HTTP status или production/staging observation.
8. Разделяй статусы:
   - `IMPLEMENTED`: работает по цепочке и подтверждено тестом или воспроизводимой проверкой;
   - `PARTIAL`: работает только часть flow, есть ручные действия или известные ограничения;
   - `UI_ONLY`: есть интерфейс, но нет полноценного backend-поведения;
   - `BACKEND_ONLY`: API/БД есть, но пользовательский flow недоступен;
   - `BROKEN`: воспроизводимая ошибка;
   - `LEGACY/DEAD`: старый, неиспользуемый или противоречащий канону путь;
   - `NOT_VERIFIED`: проверить невозможно, и почему.
9. При конфликте кода, UI и документации укажи все три версии и выбери фактическую source of truth с обоснованием.

## Контекст проекта, который нужно проверить

### Продуктовая идея

Kamilya LMS — корпоративная multi-tenant LMS для компаний Казахстана. Базовая идея:

```text
документы компании → AI/методолог → курс и тест → аудитория/когорта → назначение → обучение → сертификат → контроль HR
```

Компания должна получать отдельный tenant, настраивать свою оргструктуру и обучать сотрудников по курсам, созданным из внутренних документов или загруженным в формате SCORM 1.2. Система должна быть пригодна для повторяемого использования несколькими компаниями, а не только для одного демо-тенанта.

### Технологический контур

- Frontend: Next.js 14 App Router, React, TypeScript, Tailwind, локали RU/EN/KK.
- Backend: FastAPI, SQLAlchemy async, Alembic.
- Database: Supabase Postgres, pgvector, tenant-scoped data, RLS/FORCE RLS.
- Storage: Supabase Storage, bucket `Kamilya LMS`.
- Queue/worker: Upstash Redis и Celery worker на VPS.
- API: Render, production URL `https://kamilya-lms-api.onrender.com`.
- Web: Vercel, production URL `https://app.kml.kz`.
- Email: Resend, sender `Kamilya LMS <no-reply@notify.kml.kz>`.
- AI: Qwen как основной провайдер в соответствующих модулях; fallback зависит от конкретного pipeline и должен быть проверен отдельно, а не предположен.
- Дополнительные VPS-сервисы: Docling для обработки документов и WhatsApp gateway; фактическое участие в production-flow нужно подтвердить.

### Роли и предполагаемые границы

| Роль | Предполагаемая ответственность | Что особенно проверить |
|---|---|---|
| `superadmin` | Платформа: tenants, статусы, лимиты, provider keys, диагностика, коммерческий контроль | Создание/редактирование/удаление tenant, первый админ, trial, impersonation, ошибки 4xx/5xx, отсутствие tenant leakage |
| `admin`, `org_admin` | Инфраструктура tenant: команда системных пользователей, настройки, интеграции, kiosk, обзор результатов | Не видит и не редактирует курсы, тесты, обучающихся и назначения, если это не утверждённый read-only экран |
| `methodologist` | Документы, AI-генерация, курсы, тесты, штатка как источник аудитории, cohorts, правила и ручные назначения | Видит весь рабочий learning-flow без переходов в legacy/admin screens |
| `teacher` | Текущий backend alias learning-content роли; реальная отдельная модель может отсутствовать | Не считать отдельной ролью без отличающейся policy и UX |
| `student` | Мои курсы, уроки, тесты, прогресс, сертификаты, AI-помощник, surveys | Нет доступа к tenant authoring, штатке, назначениям, admin и platform routes |
| kiosk operator | Отдельный короткий kiosk-flow с токеном | Авто-logout, очистка состояния, отсутствие доступа к обычному кабинету, приватность предыдущего сотрудника |

Это ожидаемая модель, а не утверждение. Проверь реальный RBAC на frontend, backend, SQL/RLS и прямых запросах.

## Функциональные флоу для обязательной трассировки

### 1. Регистрация tenant и вход

Проверь полную цепочку:

1. HR открывает `/register-tenant`.
2. Заполняет компанию, контактное лицо, email, пароль, телефон и дополнительные сведения.
3. Создаются tenant, первый пользователь `admin`, trial plan и usage limits.
4. При необходимости отправляется приглашение/подтверждение.
5. Вход выполняется email OTP через Resend или Telegram code flow.
6. После входа восстанавливается refresh-сессия после F5 и пользователь попадает именно в tenant dashboard.
7. Сессия не смешивает tenant и student role.

Проверь нейтральные ответы для неизвестного email, cookie `HttpOnly`, CORS, refresh/logout, повторную регистрацию, duplicate email/slug, валидацию телефона и обработку Resend failure.

### 2. Tenant administration и superadmin

Проверь:

- список tenant-ов и фактические counters;
- создание tenant в одном мастере: компания → первый админ → trial/лимиты → invitation;
- автоматический slug и уникальность;
- редактирование и удаление tenant, каскады и ограничения;
- смену статуса, trial end date, tariff и лимитов;
- отображение tenant name в кабинете без дублирования и обрезания;
- разделение global provider configuration и tenant integrations;
- прямой доступ к `/admin/super/*` обычным tenant-пользователем;
- impersonation и audit trail, если заявлены;
- понятные ошибки вместо сырых 500.

Отдельно зафиксируй, какие коммерческие функции реально отсутствуют: billing, upgrade, invoices, payment webhooks, trial enforcement, lead pipeline, quota enforcement.

### 3. Методолог: документ → курс → тест

Проверь рабочий сценарий:

1. Загрузка документов (`md`, `txt`, `pdf`, `docx`, при заявленной поддержке другие форматы).
2. Реальный upload, progress, timeout/cancel, индексирование, retry и объяснение ошибки.
3. Выбор документов, языка курса и количества модулей.
4. Генерация курса из одного источника на русском и казахском либо точное описание фактической модели локализации.
5. AI fallback, лимиты, очередь, повторная генерация и зависшие jobs.
6. Результат: курс, модули, уроки, тесты, статусы draft/published.
7. Ручное редактирование курса и тестов.
8. Публикация только валидного курса.
9. Идемпотентность при повторном клике/refresh.

Не считать наличие `AI generation` экраном доказательством готовности пайплайна. Укажи, где результат хранится, как связаны документы с курсом и как обрабатываются частичные AI-ответы.

### 4. Штатка, сотрудники, обучающиеся и cohorts

Проверь, не смешиваются ли:

- системные пользователи tenant (`admin`, `methodologist`, `teacher`);
- сотрудники/штатные единицы;
- обучающиеся (`student`);
- приглашённые, активные, заблокированные и удалённые записи.

Для импорта штатки проверь:

- `xlsx` с несколькими листами;
- выбор листа и заголовка;
- сопоставление колонок по русским/английским вариантам;
- preview до записи;
- обязательные и необязательные поля;
- дубли, пустые строки, ошибки отдельных строк;
- обновление существующих сотрудников без дублирования;
- понятную причину ошибки и возможность исправить её в UI.

Для ручного добавления сотрудника проверь, что это находится в зоне методолога и не превращается в создание системного пользователя.

### 5. Назначения, cohorts и автоматические правила

Проверь источники назначений:

- `department`;
- `position`;
- `cohort`;
- `manual`.

Нужно установить:

1. где создаётся правило;
2. когда выполняется пересчёт;
3. выполняется ли он sync или Celery worker;
4. что происходит при добавлении нового сотрудника в cohort;
5. что происходит при удалении/изменении правила;
6. как предотвращаются дубли enrollment;
7. как хранится `source` и кто может удалить назначение;
8. что видит обучающийся после назначения;
9. отправляется ли приглашение автоматически или вручную;
10. где ссылка на курс/тест и можно ли скопировать/отозвать её.

Сравни UI, API, модели, worker и фактический результат в БД.

### 6. Learner flow и сертификат

Проверь end-to-end:

1. обучающийся получает invite link или другой доступ;
2. принимает приглашение и создаёт/подтверждает аккаунт;
3. после входа видит только свои назначения;
4. открывает курс, уроки и тесты;
5. прогресс сохраняется после reload и при повторном запросе;
6. обязательные уроки и quiz действительно влияют на completion;
7. 404/409/500 на quizzes и course completion корректно обработаны;
8. повторная отправка completion идемпотентна;
9. сертификат создаётся один раз, доступен в UI и PDF можно скачать;
10. запись появляется в training log и доступна разрешённой роли;
11. после завершения корректно работает AI assistant/survey, если они заявлены.

Проверь отдельно, что студент не может подменить `course_id`, `user_id`, tenant или результат теста.

### 7. SCORM 1.2

Проверь фактический scope: поддерживается ли именно SCORM 1.2, а SCORM 2004 корректно отклоняется.

Обязательные сценарии:

- manifest и launch resource;
- вложенные assets, query string и hash;
- iframe/player;
- LMSInitialize, LMSSetValue, LMSCommit, LMSFinish;
- completion, progress и повторное открытие;
- storage/security/path traversal/XSS;
- отсутствие дубликата сертификата;
- реальные iSpring/Articulate ZIP на staging, если доступны.

Unit parser tests без browser/storage/database E2E не считать доказательством полного SCORM flow.

### 8. Kiosk

Проверь, существует ли рабочий kiosk как самостоятельный продуктовый surface, а не только пункт меню:

- создание устройства/ссылки/токена;
- привязка к tenant;
- идентификация сотрудника;
- запуск назначения;
- авто-logout и очистка browser state;
- отсутствие доступа к данным предыдущего сотрудника;
- завершение и сертификат;
- отзыв/ротация токена;
- мобильный и desktop UX.

### 9. Trial, лимиты и billing

Заявленный trial: 14 дней, 1 обычный AI-курс, 1 курс по должностной инструкции, 10 обучающихся, 3 системных пользователя. Проверь не только карточки dashboard, но и enforcement на backend:

- создание второго курса;
- генерация сверх quota;
- добавление сотрудника/обучающегося сверх лимита;
- публикация/импорт сверх лимита;
- истёкший trial;
- concurrent requests/race conditions;
- единая ошибка с объяснением и CTA;
- расходование лимита при failed/retried job;
- superadmin override и аудит изменений.

Если UI показывает `0/1`, а endpoint продолжает создавать второй объект, это критическое расхождение.

### 10. Инфраструктура и эксплуатационная применимость

Проверь:

- health/readiness endpoints;
- Render/Vercel deploy source of truth и commit parity;
- Celery worker и Redis queue;
- Docling и storage;
- database connection split `DATABASE_URL` vs `MIGRATION_DATABASE_URL`;
- migrations head и rollback risk;
- RLS/FORCE RLS и application tenant filters;
- логирование и correlation IDs;
- retry/timeouts/dead letter для AI/email/worker;
- backup/restore и recovery assumptions;
- rate limits, file limits, upload DoS, SSRF/path traversal/XSS;
- observability, alerts, error budget и ручные operational steps.

Не выдавай deployment health `200` за доказательство бизнес-готовности.

## Формат итогового отчёта

Сохрани результат в отдельном Markdown-файле в ветке аудита. Не изменяй `master`.

### 1. Executive verdict

В начале дай:

- `GO`, `GO WITH CONDITIONS` или `NO-GO` для первого production-тенанта;
- что именно блокирует запуск;
- что можно оставить полуручным;
- что нельзя оставлять полуручным из-за риска потери данных, tenant leakage, неверных сертификатов, auth или billing.

### 2. Матрица функций

Таблица:

| Домен | Функция | Роль | Статус | Evidence | Риск | Минимальный fix | Приоритет |
|---|---|---|---|---|---|---|---|

### 3. Матрица ролей и маршрутов

Для каждой роли: разрешённые маршруты, запрещённые маршруты, прямой API-доступ, фактический UI, расхождение.

### 4. Матрица флоу

Для каждого flow: шаг, frontend, endpoint, DB/worker, expected result, actual result, failure mode, status.

### 5. Несоответствия docs vs implementation

Формат:

| Документ/утверждение | Фактическая реализация | Доказательство | Как исправить документацию или код |
|---|---|---|---|

### 6. Findings

Каждое finding оформляй так:

```text
ID:
Severity: P0/P1/P2/P3
Title:
Affected role/flow:
Reproduction:
Expected:
Actual:
Evidence:
Root cause:
Impact:
Minimal remediation:
Regression test:
```

Severity:

- `P0`: запуск блокируется, auth/RLS/tenant isolation/data integrity/security/неверная выдача сертификата;
- `P1`: критичная рабочая функция первого tenant-flow не работает или вводит пользователя в заблуждение;
- `P2`: ограничение, плохой UX, ручная операция или слабое покрытие без немедленного риска;
- `P3`: косметика, документация или долгосрочное улучшение.

Не называй проблему P0 без воспроизводимого воздействия.

### 7. План исправлений

Составь последовательность:

1. обязательные P0 до запуска;
2. P1 для стабильной эксплуатации первого tenant;
3. P2 для масштабирования и UX;
4. что не делать сейчас.

Для каждого пункта укажи owner boundary, затронутые файлы/модули, миграции, тесты, ручную QA-проверку и критерий готовности.

### 8. Итоговая оценка применимости

Оцени отдельно по шкале `0–5` с обоснованием:

- product completeness;
- implementation integrity;
- role/RBAC correctness;
- tenant isolation;
- auth/session reliability;
- learning delivery;
- AI/document pipeline;
- SCORM readiness;
- certificate/training records;
- trial/billing enforcement;
- operational readiness;
- test confidence;
- UX clarity.

Финальный текст должен отвечать на вопрос: «Можно ли дать систему первому реальному тенанту, какие функции он сможет выполнить без вмешательства разработчика, а где потребуется ручной оператор?»

## Запрещённые выводы

- Не писать «проект выглядит хорошо» без измеримого критерия.
- Не считать наличие страницы, кнопки или endpoint полноценной реализацией.
- Не считать зелёный deploy доказательством готовности.
- Не считать unit-тест parser-а доказательством end-to-end готовности SCORM.
- Не считать dashboard counters доказательством enforcement лимитов.
- Не считать frontend role guard заменой backend authorization и RLS.
- Не скрывать незапущенные проверки под формулировкой «не удалось проверить» без причины и оценки риска.
- Не предлагать большой рефакторинг без связи с конкретным finding.

## Критерий хорошего результата

После отчёта руководитель должен без чтения исходного кода понимать:

1. какие части Kamilya LMS реально работают;
2. какие работают только в демо или вручную;
3. где описание продукта расходится с кодом;
4. что блокирует первого tenant;
5. какой минимальный план исправлений нужен;
6. какие утверждения можно безопасно использовать во внешней презентации, а какие нельзя.

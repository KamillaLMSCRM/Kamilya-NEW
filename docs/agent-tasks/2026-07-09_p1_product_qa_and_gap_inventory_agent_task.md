## Роль агента

Ты внешний агент-аналитик/QA-помощник. Твоя задача - выполнить большую, трудоемкую, но низкорисковую работу: пройти продукт, собрать факты, описать UX/UI-проблемы, сравнить с целевой логикой LMS и подготовить структурированный backlog P1.

## Обязательное правило ветки

## Главный продуктовый контекст

Kamilya LMS - не копия Chamilo. Это HR-first LMS для корпоративных tenant. Главный целевой поток:

`регистрация tenant -> первый админ -> штатка/сотрудники -> должности -> документы/ДИ -> AI-курс -> тест -> назначение -> прохождение обучающимся -> сертификат -> журнал/отчет`

Chamilo использовать как ориентир зрелости LMS, а не как требование копировать все подряд.

### 2. Flow-аудит приглашений и назначений

Это важный P1-блок. Нужно понять и описать, как должно работать назначение курса и как работает сейчас.

Проверь и опиши:

- назначение курса конкретному сотруднику;
- назначение по должности;
- назначение по отделу;
- массовое назначение;
- повторная отправка приглашения;
- ссылка приглашения для обучающегося;
- что видит пользователь без аккаунта;
- что видит пользователь с аккаунтом;
- как обучающийся попадает в курс;
- как назначение связано с тестом;
- что происходит после дедлайна;
- где tenant admin/methodologist видит статус.

Если в UI нет явной ссылки или кнопки приглашения - зафиксируй это как gap. Не додумывай, что оно "наверное есть в API".

### 3. Methodologist workspace

Проверь, что разделы методолога логически связаны между собой:

- документы;
- AI-генерация курса;
- курсы;
- конструктор тестов;
- должности;
- штатка;
- назначения;
- сертификаты.

Особенно проверь:

- где методолог создает курс;
- где выбирает язык курса;
- можно ли из одного источника сделать RU/KZ версии;
- где создается/редактируется тест;
- как тест привязан к уроку/курсу;
- как назначается курс;
- где видно, что курс готов к публикации;
- где видны ошибки генерации/индексации.

Результат: список UX-разрывов и предложенный порядок экранов/CTA.

### 4. Tenant admin workspace

Проверь, что tenant admin не смешан с обучающимся и методологом.

Пройти:

- вход после регистрации trial;
- видимость названия tenant;
- лимиты trial;
- пользователи системы;
- сотрудники/обучающиеся;
- импорт штатки;
- ручное добавление сотрудника;
- управление ролями;
- отчеты;
- настройки сертификатов;
- интеграции уведомлений;
- billing/plan placeholders.

Особенно отметить:

- где перепутаны "пользователи системы", "сотрудники", "обучающиеся";
- где кнопки ведут не туда;
- где текст старый или технический;
- где обрезаются названия в sidebar;
- где нет empty/error/loading state.

### 5. Superadmin workspace

Проверить:

- список tenant;
- создание tenant через wizard;
- slug auto-generation;
- первый admin;
- trial limits;
- edit tenant;
- delete tenant;
- tenant detail;
- AI providers;
- notification integrations.

Задача агента: не писать новый backend, а собрать точный UX/QA backlog и мелкие UI-исправления, если безопасно.

### 6. SCORM 1.2 P1 UX

Не нужно расширять до SCORM 2004.

Проверь:

- где импортируется SCORM;
- понятны ли ограничения "только SCORM 1.2";
- как показывается ошибка SCORM 2004;
- есть ли preview/launch после импорта;
- видно ли, что курс SCORM, а не обычный AI-курс;
- сохраняется ли статус;
- как это отображается в learner dashboard;
- как это попадает в training log.

Результат: рекомендации по UI и отчетным статусам.

### 7. Kiosk flow

Проверь и опиши:

- где tenant admin/methodologist создает kiosk;
- как генерируется ссылка/QR;
- что видит сотрудник на общем устройстве;
- как выбирается/идентифицируется сотрудник;
- как стартует курс;
- как фиксируется прохождение;
- что происходит при ошибке/обрыве;
- как админ видит kiosk activity.

Если kiosk пока foundation-level, честно указать, что именно есть, а чего не хватает для реального цеха/склада.

### 8. Learner AI assistant

Проверь:

- где ассистент появляется в course player;
- знает ли контекст курса/урока;
- есть ли ограничения по ответам;
- понятно ли пользователю, что это помощник по материалу курса;
- что происходит при ошибке AI provider;
- есть ли fallback;
- сохраняется ли история.

Сделай рекомендации: что нужно для P1, что оставить на P2.

### 9. Отчеты, журнал, сертификаты

Проверь:

- training log;
- статусы assigned/in_progress/completed/failed/expired;
- выдачу сертификата;
- шаблон сертификата;
- скачивание/просмотр;
- экспорт отчетов;
- связь сертификата с курсом/тестом/сотрудником;
- насколько это понятно tenant admin.

Сформируй таблицу недостающих отчетов для первого платного tenant.

### 10. Mobile/tablet UI

Сделай ручной QA или Playwright screenshots по ширинам:

- 390px mobile;
- 768px tablet;
- 1440px desktop.

Проверить минимум:

- login/register;
- admin dashboard;
- staff import/manual add;
- AI generation;
- courses;
- quiz builder;
- learner dashboard;
- course player;
- superadmin tenants.

Фиксировать:

- horizontal scroll;
- overlapping text;
- clipped sidebar labels;
- unreachable buttons;
- dialogs wider than viewport;
- tables unusable on mobile.

### 11. Тексты и i18n

Собери список:

- hardcoded русский/английский;
- неправильные термины;
- старые legacy-названия;
- технические сообщения ошибок;
- странные CTA.

Где безопасно, можно исправить тексты прямо в коде. Но не менять продуктовую логику без согласования.

Предпочтительная терминология:

- "тенант" во внутренних документах, но в UI для клиента чаще "компания" или "кабинет компании";
- "сотрудники" для штатки;
- "пользователи системы" для admin/methodologist/superadmin;
- "обучающиеся" для тех, кому назначают курсы;
- "назначение курса", не "enrollment" в UI;
- "шаблон сертификата", не "настройки сертифи..." в sidebar.

Можно делать:

- менять AI provider routing;
- делать destructive cleanup;
- merge в `master`.

## Как работать с тестами

От агента не требуется глубокая разработка тестов. Но требуется аккуратно проверять, что он не сломал проект.

Минимум перед финалом:

```bash
cd apps/api
python -m pytest tests/unit -q
python -m compileall app
```

```bash
cd apps/web
npm run typecheck
```

Если окружение не позволяет запустить команду, укажи причину в отчете.

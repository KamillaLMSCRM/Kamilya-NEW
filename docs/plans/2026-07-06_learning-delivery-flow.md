# План доработки flow приглашений на курс и тест

Дата: 2026-07-06
Статус: отложено, вернуться позже

## Цель

Собрать единый flow: методолог назначает курс или тест, система понимает, есть ли у сотрудника активный доступ, дает ссылку или отправляет приглашение, а методолог видит статус доставки и прохождения.

## 1. Разделить сущности

`Assignment` - факт назначения курса или теста сотруднику.

Уже есть:
- `Enrollment` для курса;
- `QuizAssignment` для теста.

`Access invite` - приглашение сотрудника в кабинет: `/accept-invite?token=...`. Нужно, если сотрудник еще не активировал аккаунт.

`Learning link` - ссылка на конкретную учебную сущность:
- курс: `https://app.kml.kz/courses/{courseId}`;
- тест: `https://app.kml.kz/courses/quiz/{quizId}`.

## 2. Backend: delivery service

Добавить сервис условно `learning_delivery_service`.

Метод:

```text
prepare_learning_delivery(user_id, target_type, target_id)
```

Логика:
- если пользователь активен и имеет способ входа, вернуть direct learning link;
- если пользователь inactive/pending и есть email, создать или переиспользовать invite-link;
- если нет email, вернуть статус `missing_contact`;
- если курс/тест не назначен этому пользователю, вернуть ошибку доступа.

Ответ:

```json
{
  "user_id": "...",
  "target_type": "course|quiz",
  "target_id": "...",
  "access_status": "active|invite_required|missing_contact",
  "learning_url": "...",
  "invite_url": "...",
  "delivery_status": "not_sent|sent|failed"
}
```

## 3. Backend: отправка email

Добавить endpoint:

```text
POST /v1/learning-delivery/send
```

Body:

```json
{
  "target_type": "course",
  "target_id": "...",
  "user_ids": ["..."],
  "channel": "email"
}
```

Для каждого пользователя:
- проверить назначение;
- подготовить ссылку;
- если активен, отправить email с кнопкой "Перейти к курсу/тесту";
- если не активен, отправить email с кнопкой "Принять приглашение и пройти курс/тест";
- сохранить delivery log.

## 4. DB: журнал доставок

Добавить таблицу `learning_deliveries`:

- `id`
- `tenant_id`
- `user_id`
- `target_type`
- `target_id`
- `channel`
- `status`: `prepared | sent | failed | opened | accepted`
- `invite_id`, nullable
- `learning_url`
- `error`
- `sent_at`
- `created_at`

Назначение таблицы: методолог должен видеть фактический статус доставки, а не только факт назначения.

## 5. UI: курсы

В `/assignments` после назначения курса:
- показывать список назначенных сотрудников;
- рядом действия:
  - `Скопировать ссылку`;
  - `Отправить приглашение`;
  - `Отправить всем выбранным`;
- показывать статус по каждому сотруднику:
  - `Активен`;
  - `Нужно приглашение`;
  - `Нет email`;
  - `Отправлено`;
  - `Начал`;
  - `Завершил`.

## 6. UI: тесты

В `/quizzes` в панели назначения:
- после назначения теста показывать delivery-блок;
- для каждого назначенного сотрудника:
  - `Скопировать ссылку на тест`;
  - `Отправить приглашение`;
  - статус прохождения.

Важно: не смешивать "назначить тест" и "отправить приглашение". Назначение создает право/обязанность пройти. Приглашение доставляет доступ.

## 7. UI: карточка сотрудника в штатке

В `/staff` в структуре сотрудника добавить компактное действие:
- `Пригласить в кабинет`, если пользователь inactive и есть email;
- `Скопировать invite`, если email нет или нужна ручная отправка;
- позже: `Открыть прогресс`.

## 8. Email-шаблоны

Минимум 3 шаблона через Resend:

`account_invite`
- "Вас пригласили в Kamilya LMS";
- кнопка принять приглашение.

`course_assigned`
- "Вам назначен курс";
- название курса;
- кнопка открыть курс.

`quiz_assigned`
- "Вам назначен тест";
- название теста;
- дедлайн, если есть;
- кнопка открыть тест.

Если пользователь еще не активен, письмо должно вести через invite-link, а после принятия приглашения редиректить к курсу/тесту.

## 9. Redirect after invite

Доработать invite URL.

Сейчас:

```text
/accept-invite?token=...
```

Нужно:

```text
/accept-invite?token=...&next=/courses/{courseId}
/accept-invite?token=...&next=/courses/quiz/{quizId}
```

После установки пароля пользователь сразу попадает на нужный курс или тест.

## 10. Приоритет реализации

1. Backend delivery prepare/send для курса.
2. Email через Resend для курса.
3. UI в `/assignments`: копировать ссылку / отправить email.
4. Поддержка `next` в `/accept-invite`.
5. То же самое для тестов в `/quizzes`.
6. Delivery log и статусы в UI.
7. Кнопки в `/staff`.

## MVP-версия

Самый быстрый полезный вариант:

- добавить `Скопировать ссылку` и `Отправить email` после назначения;
- если сотрудник inactive, отправлять invite-link с `next`;
- если active, отправлять direct-link;
- если нет email, показывать `Нет email` и кнопку копирования.

Это закрывает первый продовый сценарий без сложной автоматизации.

Дата: 2026-07-24

Основание:

## Цель

Собрать кабинет методолога вокруг последовательного рабочего процесса:

```text
источник
  -> курс или тест
  -> сотрудники и квалификации
  -> аудитория и правило назначения
  -> прохождение
  -> результат и обратная связь
```

После доработки методолог не должен искать одну операцию в нескольких
технических разделах, работать через исторические `/admin/*` URL или
сталкиваться с разными UI для одного предметного объекта.

Tenant admin по-прежнему не управляет курсами, тестами, учащимися и
назначениями. Его область: команда администраторов, настройки tenant-а,
интеграции, киоски и шаблон сертификата.

## Правила исполнения

1. Работа выполняется волнами. Следующая волна начинается только после review и
   merge предыдущей.
2. Каждая исполнительская задача работает в отдельном Codex worktree и новой
   ветке `agent/methodologist-*`.
3. Исполнитель не меняет production, Render, Vercel, Supabase, VPS и секреты.
4. Не возвращать legacy-роль `teacher`; каноническая роль контента только
   `methodologist`.
5. Не расширять доступ tenant admin к учебным функциям.
6. Не создавать второй API или второй UI для уже существующей операции.
7. Любое изменение API сопровождается backend-тестом, изменение policy или
   navigation — frontend-тестом.
8. Исполнитель завершает задачу коммитом в своей ветке и сообщает SHA, список
   изменённых файлов, команды проверок и известные ограничения.
9. Интегратор независимо читает diff, запускает проверки и только после этого
   переносит изменения в `master`.

## Волна 0 — подтверждённые дефекты и локальные UX-проблемы

Волна делится между двумя независимыми исполнителями. Они не должны менять
одни и те же файлы, кроме i18n-файлов; изменения локализаций при интеграции
проверяются отдельно.

### Задача A — функциональные контракты

Ветка: `agent/methodologist-functional-contracts`

#### A1. Cohorts получает реальный список пользователей

Файлы:

- `apps/web/src/app/cohorts/page.tsx`;
- новый или существующий frontend-тест для разбора `UserListResponse`.

Изменение:

- использовать контракт `{ users, total, page, per_page }`;
- убрать fallback на несуществующее `items`;
- типизировать минимальную модель пользователя без `any`;
- отделить `error` от успешного пустого списка.

Критерии:

- участники отображаются при `{ users: [...] }`;
- пустое состояние появляется только после успешного ответа;
- ошибка загрузки не выглядит как «участников нет».

#### A2. Один канонический flow назначения тестов

Файлы:

- `apps/web/src/app/admin/quizzes/assign/page.tsx`;
- при необходимости route-тест.

Изменение:

- standalone-страница не должна повторять `QuizAssignPanel`;
- сделать redirect на `/quizzes` с понятным query-параметром или anchor,
  открывающим вкладку назначения;
- не исправлять второй form-state и не сохранять дублирующий UI.

Критерии:

- старый URL не падает и ведёт в канонический конструктор;
- пользователь не видит две разные формы назначения теста.

#### A3. Честная рассылка «Весь tenant»

Файлы:

- `apps/api/app/modules/announcements/router.py`;
- backend-тесты announcements.

Изменение:

- без `course_id` выбирать всех активных пользователей текущего tenant-а с
  непустым email, без обязательного `JOIN Enrollment`;
- с `course_id` выбирать только активных участников этого курса;
- сохранять дедупликацию email;
- не допускать получателей другого tenant-а.

Критерии:

- пользователь без enrollment получает tenant-wide объявление;
- course announcement не уходит пользователю без enrollment на курс;
- inactive, email-less и cross-tenant пользователи исключены.

#### A4. Исправить подпись выбора курса

Файл:

- `apps/web/src/features/course-assignments/CourseAssignmentsPage.tsx`;
- i18n `ru/en/kk`, если текст не вынесен.

Изменение:

- placeholder описывает выбор курса, а не статус;
- количество курсов не маскируется под количество статусов.

## Gate 0 — интеграционная проверка волны 0

Ответственный: основной агент-интегратор.

1. Прочитать каждый diff и сопоставить его с задачей.
2. Убедиться, что ветки не расширяют роли и не затрагивают tenant isolation.
3. Запустить объединённые backend/frontend tests.
4. Проверить миграции: в волне 0 их быть не должно.
5. Провести локальный browser smoke:
   - cohorts показывает людей;
   - старый quiz assignment URL ведёт в канонический flow;
   - courses не переполняется;
   - training log читаем на mobile;
   - announcements не отправлять live, проверять тестом.
6. Merge выполняется в порядке A -> B.

Gate пройден, когда все критерии подтверждены кодом и тестами. Только после
этого начинается волна 1.

## Волна 1 — единый route/capability registry и новая навигация

Ветка: `agent/methodologist-navigation-registry`

Эта задача не запускается параллельно с волной 0.

### C1. Capability model

В `apps/web/src/lib/` создать единый типизированный registry:

```text
manage_content
manage_staff
manage_qualifications
manage_assignments
view_training_log
manage_communications
configure_tenant
learn
manage_platform
```

Registry хранит:

- канонический route;
- capability;
- label key;
- icon key или компонентный mapping;
- section;
- порядок;
- видимость в Sidebar/Command Palette;
- legacy redirects отдельно от navigation.

`Sidebar`, `CommandPalette`, `rolePolicy` не должны независимо кодировать одну и
ту же карту доступов.

### C2. Новые группы меню

Реализовать целевую IA из этого плана. Для methodologist обязательно добавить
`/dashboard` первым пунктом.

Запрещено:

- показывать tenant admin учебные пункты;
- возвращать learner shortcuts методологу только ради demo;
- помещать сертификатный шаблон в группу курсов.

### C3. Канонические URL

Создать канонические маршруты:

```text
/staff
/invitations
/training-log
/quizzes/assignments
```

Старые:

```text
/admin/staff
/admin/invitations
/admin/training-log
/admin/quizzes/assign
```

должны стать redirects или совместимыми thin wrappers. Новый код и навигация
ссылаются только на канонические URL.

### C4. Тесты

- таблица role x capability;
- доступность всех navigation items;
- соответствие Sidebar и Command Palette одному registry;
- legacy redirect tests;
- methodologist имеет dashboard;
- admin не получает content/staff/assignment capability;
- student не получает authoring routes;
- superadmin не получает tenant learning routes автоматически.

### Gate 1

- нет нового дублирования массивов route/role;
- все role policy tests и navigation E2E зелёные;
- desktop/mobile menu читаемо;
- прямые старые ссылки не дают 404;
- product owner принимает названия групп.

## Волна 2 — устранение продуктового дублирования

### D1. Единая библиотека источников

- `/documents` становится каноническим менеджером;
- категории «Общие документы» и «Должностные инструкции»;
- отображение использований источника;
- защищённое удаление связанного источника;
- AI wizard использует selector существующих источников и compact upload, но не
  второй файловый менеджер.

Перед реализацией требуется отдельная схема API и UX-прототип.

### D2. Карточка должности как агрегат квалификации

Вкладки:

1. профиль;
2. должностная инструкция;
3. компетенции;
4. обязательное обучение;
5. onboarding-тест;
6. история версий.

`/staff?tab=rules` не редактирует повторно position-course links, а показывает
их read-only с переходом в должность.

### D3. Assignment Hub

Единый экран должен объяснять назначение:

- кому;
- что;
- на каком основании;
- когда назначено;
- дедлайн;
- состояние применения правила.

Источники сохраняются как разные backend-механизмы: manual, position,
department, cohort, company. UI объединяет наблюдение и запуск, но не стирает
происхождение.

### D4. Постоянное company-wide правило

Требуется ADR и миграция. Новые отделы и сотрудники должны наследовать правило
без повторного batch action. Нужны idempotency, история изменений и тесты
tenant isolation.

### Gate 2

- один канонический редактор каждого отношения;
- UI показывает источник назначения;
- нет скрытого snapshot поведения;
- миграции проходят на чистой и существующей БД;
- tenant isolation tests обязательны.

## Волна 3 — дизайн-система и accessibility

1. Ввести semantic tokens для `success`, `warning`, `danger`, `muted`,
   interactive border с AA-контрастом текста.
2. Зафиксировать:
   - surface radius `12px`;
   - control radius `8px`;
   - pill только для badges/chips;
   - один surface на смысловой блок.
3. Создать единый `Dialog`:
   - focus trap;
   - Escape;
   - `aria-labelledby`;
   - mobile padding `16px`;
   - `max-height: calc(100dvh - 32px)`;
   - sticky action footer.
4. Добавить варианты `Table bare` и `TableContainer`.
5. Провести responsive QA ключевых authoring screens.
6. Добавить автоматические a11y smoke tests и визуальные screenshots.

### Gate 3

- WCAG AA для обычного текста и статусов;
- все диалоги доступны с клавиатуры;
- нет вложенных cards только ради рамки;
- нет горизонтального scroll страницы на `390px`;
- Playwright screenshots проходят на трёх размерах.
